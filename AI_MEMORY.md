# AI Memory

AIが開発作業を始める前に確認する、ユーザー固有の開発方針、作業ルール、報告形式です。このファイルは秘密情報や会話履歴の保存場所ではありません。長期的に必要な作業方針だけを記録します。

## 開発方針

- 正式ソース以外は変更しない。
- 正式ソースは `C:\Users\suisy\Documents\Development\repos` 配下とする。
- 旧フォルダは参照専用とし、新規開発、修正、ビルド、コミットに使用しない。
- 調査と実装を分離する。依頼が調査だけの場合は、コードや設定を変更しない。
- 不明なファイル、設定、データを推測で削除しない。
- 業務ロジックを明示的な合意なく変更しない。
- 既存の未コミット変更はユーザーの作業として維持し、上書き、破棄、巻き戻しをしない。
- コミット、タグ、pushは明示的な指示があるまで行わない。
- 実装後は、変更内容、確認結果、未確認項目、Git差分を報告する。
- 実運用設定、秘密情報、認証情報、顧客データ、業務データをGit管理しない。
- 新しい設計判断や運用ルールは `development-management` に記録する。
- チャットだけを唯一の情報源にしない。GitHub上の`development-management`と対象リポジトリを確認する。
- 推測と確認済み事実を分け、未確認事項は「未確認」と明記する。

## 調査依頼と実装依頼の境界

### 調査依頼

- 読み取り、Git状態確認、ログ確認、原因特定、影響範囲の整理まで行う。
- コード、設定、業務データ、外部サービスの状態を変更しない。
- 修正案を提示しても、実装許可がなければ変更しない。

### 実装依頼

- 明示された目的と変更範囲内で実装する。
- 既存の未コミット変更と重なる場合は内容を確認し、維持したまま作業する。
- 業務ロジックや運用方法を変える必要が生じた場合は、理由と影響を示して確認する。
- 実装と検証後、関連するREADME、`projects/*.md`、`PROJECT_STATUS.md`、`CHANGELOG.md`等を更新する。

## 作業の進め方

1. 現状調査 — 正式ソース、ブランチ、タグ、未コミット変更、関連文書を確認する。
2. 原因特定 — 事実と推測を分け、再現条件と影響範囲を明確にする。
3. 実装方針提示 — 変更対象、変更しない領域、確認方法を示す。
4. 実装 — 合意された範囲だけを変更し、既存変更を維持する。
5. テスト — 構文確認、自動テスト、手動起動など変更に応じた確認を行う。
6. 実運用確認 — 配布先や実機で必要な確認を行い、できない場合は未確認とする。
7. 進行状況更新 — 管理文書と対象プロジェクトのREADMEへ結果を反映する。
8. コミット可否確認 — コミット、push、タグ作成は明示的な許可を得てから行う。

## 回答・報告の好み

- 結論を先に伝える。
- 追加情報を無制限に広げず、必要な作業をまとめて提示する。
- 指示文は、そのまま貼り付けて使える完成形で提示する。
- 何を、どのチャットまたはどのプロジェクトへ貼るかを明確にする。
- できたことと未確認事項を分ける。
- 推測と確定事項を混ぜない。
- コードや文書を変更した場合は、対象ファイルと確認結果を明示する。
- 作業を行っていない対象について、変更済み・確認済みと表現しない。

## プロジェクト運用

新しいチャットでは、`development-management`を最初に確認し、次の順序で読む。

1. [README.md](README.md)
2. [PROJECT_STATUS.md](PROJECT_STATUS.md)
3. [AI_STARTUP.md](AI_STARTUP.md)
4. 本ファイル（`AI_MEMORY.md`）
5. [docs/ai_handoff.md](docs/ai_handoff.md)
6. 対象プロジェクトの`projects/*.md`
7. 必要に応じて[VERSION_MATRIX.md](VERSION_MATRIX.md)、関連する管理文書、対象リポジトリ本体

対象リポジトリを確認するときは、READMEだけでなく、現在のブランチ、最新タグ、最新コミット、未コミット変更も確認する。

## 作業完了時の最低報告項目

- 変更した内容とファイル
- 実施した確認と結果
- 実運用で未確認の項目
- 既存変更を含む現在のGit差分
- `development-management`へ記録した判断・進行状況
- コミット、push、タグの実施有無
